# Urinavi 構成まとめ

## 1. アプリ概要

### アプリ名
**Urinavi（ウリナビ）**

### 名前の意味
「売り場をナビするアプリ」という意味。  
お客さん・従業員が商品を探すときに、店内のどこにあるかをすぐ確認できるアプリを目指す。

### 目的
Urinaviは、スーパーや小売店舗で「商品がどこにあるか分からない」という問題を解決するための店内商品検索アプリ。

お客さん側は、商品名やカテゴリから売り場を探せる。  
従業員側は、バーコード検索や商品情報の確認によって、品出し・売り場戻し・案内作業を効率化できる。

---

## 2. 解決したい課題

### お客さん側の課題
- 欲しい商品がどの通路にあるか分からない
- 店員に聞くのが面倒、または聞きづらい
- 店員を探す時間がかかる
- 売り場に商品があるのか、そもそも取り扱いがあるのか分からない
- 店内マップがないと、通路番号だけでは場所をイメージしにくい

### 従業員側の課題
- お客さんから商品の場所を聞かれるたびに作業が止まる
- 新人やアルバイトは商品の場所を覚えるまで時間がかかる
- 棚番がない店舗では、正確な案内が難しい
- お客さんが購入をやめた商品を売り場に戻すとき、場所を探す時間がかかる
- 納品・品出し作業中に、商品の置き場所確認が必要になる

### 店舗側の課題
- 接客対応に時間が取られ、品出しやレジ応援の効率が落ちる
- 商品案内の品質が従業員の経験に依存する
- 商品場所の共有が属人的になりやすい
- 売り場変更時に情報共有が遅れる

## 3. 現場前提

### 棚番について
ユーザーのバイト先には**棚番システムがない**。  
そのため、Urinaviでは「棚番あり前提」の設計にはしない。

### 店内の位置管理方針
生鮮食品とドラッグストア以外は、各通路に天井から吊り下げられた**通路番号**がある。  
そのため、商品位置は以下のように管理する。

```text
通路番号 + 売場名 + 目印 + 店内マップ上の位置
```

例：

```text
通路：5番通路
売場：調味料売場
目印：醤油・みりんの近く
マップ位置：中央右側エリア
```

### 位置情報の考え方
棚番号ではなく、実際の店舗で使いやすい表現を優先する。

- 通路番号
- 売場名
- コーナー名
- 近くの商品
- 目印
- 店内マップ上のエリア
- 生鮮、冷凍、日配、ドラッグストアなどの大分類

## 4. Urinaviの価値

### お客さんにとってのメリット
- 商品を探す時間が短くなる
- 店員に聞かなくても場所が分かる
- 店内マップで直感的に売り場へ行ける
- 「売り場にある / 売り場にない」が分かる
- 買い物ストレスが減る

### 従業員にとってのメリット
- 商品案内の時間を短縮できる
- 新人でも商品場所を案内しやすい
- 売り場戻し作業が楽になる
- 納品・品出し中の場所確認が早くなる
- 店舗知識の差をアプリで補える

### 店舗・企業側にとってのメリット
- 接客品質が安定する
- 作業効率が上がる
- 従業員教育の負担が減る
- お客さん満足度が上がる
- 店舗DXの小さな一歩として導入しやすい

### Win-Win構造
Urinaviは、お客さん側と企業側の両方にメリットがある。

```text
お客さん：探しやすい・聞かなくていい・買い物が早い
従業員：案内が楽・作業効率アップ・新人でも対応しやすい
店舗側：接客品質向上・業務効率化・顧客満足度向上
```

### 目的

- 商品一覧

### 商品一覧の表示例


---

## 7.3 商品詳細画面

### 目的
商品がどこにあるかを詳しく表示する。

### 表示内容
- 商品画像
- 商品名
- 取扱状況
- 売り場にある / 売り場にない
- 通路番号
- 売場名
- 目印
- 店内マップ
- 近くの商品

### 売り場にある場合

```text
商品名：カレールー
状態：売り場にあります
通路：5番通路
売場：調味料・カレー売場
目印：レトルトカレーの向かい側
```

### 売り場にない場合

売り場にない場合は、ただ「ありません」と表示するだけではなく、ユーザーが次にどうすればいいか分かる表示にする。

```text
商品名：〇〇
状態：現在、売り場にありません
補足：この商品は現在、売り場に出ていない可能性があります。
対応：お近くの従業員にご確認ください。
```

### 売り場にない場合のUI方針
- 赤色を強く出しすぎない
- 不安を与えない文言にする
- 「在庫なし」と断定しない
- 「売り場に出ていない可能性があります」と表現する
- 従業員確認につなげる

---

## 7.4 店内マップ画面

### 目的
お客さんが直感的に売り場の場所を理解できるようにする。

### 表示内容
- 店内フロアマップ
- 現在地の目安
- 通路番号
- 売場エリア
- 検索した商品の場所
- 入口、レジ、サービスカウンターなどの目印

### マップ導入の理由
通路番号だけでは、初めて来店したお客さんには分かりにくい。  
店内フロアマップを導入することで、視覚的に場所を理解しやすくなる。

### マップで表示したい要素
- 入口
- レジ
- サービスカウンター
- 生鮮食品
- 冷蔵・日配
- 冷凍食品
- お菓子
- 飲料
- 調味料
- 日用品
- ドラッグストア
- 各通路番号

- お菓子

## 7.6 マイページについて

### 方針
初期版では、お客さん側のマイページは必須ではない。

### 理由
Urinaviの主目的は「商品をすぐ探すこと」なので、ログインやマイページを作ると利用開始までの手間が増える。

### 初期版でマイページを作らないメリット
- すぐ検索できる
- ログイン不要
- 導入しやすい
- 個人情報を扱わなくてよい
- 開発範囲を小さくできる

### 将来的に追加するなら
- よく探す商品
- お気に入り商品
- 最近見た商品
- 店舗選択履歴
- 買い物メモ

# 8. 従業員側 画面構成

## 8.1 従業員ホーム画面

### 目的
従業員が素早く検索・確認作業に入れる画面。

### 表示内容
- 担当者名
- 本日の担当 / 責番
- バーコード検索ボタン
- 商品名検索ボタン
- 売り場戻し検索
- 最近検索した商品

### 「〇〇さん」について
画面上の「〇〇さん」には、責番に対応した人物名を表示する想定。

例：

```text
責番：67番
表示：山口さん
```

---

## 8.2 本日のタスク / 未対応について

### 方針
初期版では、複雑なタスク管理は入れすぎない。  
従業員画面は、納品や売り場戻しの作業効率を上げることを優先する。

### 本日のタスクに入れられる候補
- 売り場戻しが必要な商品
- 売り場変更があった商品
- 場所確認が多い商品
- 品出し対象の商品
- 未登録の商品

### 未対応に入れられる候補
- 売り場が未設定の商品
- バーコード検索でヒットしなかった商品
- 店舗側で確認が必要な商品
- 商品場所の更新待ち

### 初期版での判断
タスク管理を入れると操作が増えるため、最初は最小限でよい。  
「検索して場所を見る」ことを中心にする。

---

## 8.3 バーコード検索画面

### 目的
従業員が商品のバーコードを読み取り、置き場所を即確認できるようにする。

### 利用シーン
- お客さんが購入をやめた商品を売り場に戻すとき
- 納品された商品の置き場所を調べるとき
- 商品名が分からないとき
- 商品パッケージだけ手元にあるとき

### 操作フロー

```text
1. バーコード検索を開く
2. 商品バーコードを読み取る
3. 商品詳細を表示
4. 商品の置き場所を確認
5. 商品を売り場に戻す / 品出しする
```

### 重要方針
操作をなるべく少なくする。  
納品作業や売り場戻しでは、作業スピードが重要なので、「完了にする」ボタンのような余計な操作は初期版では不要。

---

## 8.4 商品詳細画面（従業員側）

### 目的
従業員が商品情報と置き場所を正確に確認する。

### 表示内容
- 商品画像
- 商品名
- JANコード / バーコード
- 商品カテゴリ
- 商品の置き場所
- 通路番号
- 売場名
- 目印
- 売り場にある / 売り場にない
- 備考
- 最終更新日

### 表示例

```text
商品名：〇〇カレールー 中辛
JANコード：4900000000000
カテゴリ：食品 / 調味料 / カレー
置き場所：5番通路 カレー売場
目印：レトルトカレーの向かい側
状態：売り場にあります
```

---

## 8.5 売り場戻し用確認画面

### 目的
お客さんが「やっぱりやめとく」と言って戻す必要が出た商品を、従業員が素早く元の場所へ戻せるようにする。

### 表示名の方針
「売り場に戻す場所」ではなく、**商品がおいている場所**と表示する方が自然。

### 理由
従業員が知りたいのは「戻す作業名」ではなく、「この商品が本来どこに置いてあるか」。  
そのため、画面表示は以下の方が分かりやすい。

```text
商品がおいている場所
5番通路 / カレー売場 / レトルトカレーの向かい側
```

### 操作方針
- バーコードを読む
- 商品の置き場所を見る
- 戻す

初期版では「完了」操作は入れない。

# 9. デザイン構成

## 9.1 デザインカラー

### メインカラー
緑系。

### 理由
- スーパー・食品売場と相性が良い
- 安心感がある
- 清潔感がある
- 店内案内アプリとして自然

### カラー例

```text
メイン：#2E7D32
サブ：#66BB6A
背景：#F6FAF6
カード：#FFFFFF
文字：#1F2937
補足文字：#6B7280
注意：#F59E0B
売り場にない表示：#E57373
```

---

## 9.2 UI方針

- スマホで見やすい
- 検索バーを大きくする
- ボタンを押しやすくする
- 情報を詰め込みすぎない
- 1画面1目的にする
- カード型デザインにする
- 商品画像と場所情報を分かりやすく表示する
- 高齢者やスマホに慣れていない人でも使えるようにする

---

## 9.3 アイコン

### 方針
- 1:1の正方形アイコン
- 緑系
- 店内ナビ・地図・商品検索をイメージできるもの
- シンプルでアプリ一覧でも分かりやすいもの

### アイコンに入れたい要素
- マップピン
- 店舗 / 棚 / 商品
- ナビゲーション
- Urinaviの頭文字「U」

# 22. 注意点

## 22.1 在庫とは分けて考える

Urinaviの初期版は、在庫管理アプリではない。  
表示するのはあくまで「商品が本来置いてある場所」。

そのため、在庫数を正確に表示する必要はない。

### 表現例

```text
売り場にあります
現在、売り場に出ていない可能性があります
お近くの従業員にご確認ください
```

---

## 22.2 売場変更への対応

スーパーでは売場変更が起こる。  
そのため、商品位置は後から更新できる設計にしておく。

---

## 22.3 棚番前提にしない

現場に棚番がないため、棚番号ではなく以下を中心にする。

```text
通路番号
売場名
目印
近くの商品
マップ位置
```

---

## 22.4 操作数を増やしすぎない

特に従業員側は、作業中に使うため操作数を最小限にする。

```text
検索する → 場所を見る → 作業する
```

これを基本にする。

# 24. まとめ

Urinaviは、商品場所を探すお客さんと、商品案内や売り場戻しを行う従業員の両方を助けるアプリ。  

現場には棚番がないため、棚番号ではなく、通路番号・売場名・目印・店内マップを組み合わせた設計にする。  

最初から完璧なシステムを作るのではなく、まずはReact Nativeで試作品を作り、商品検索・店内マップ・バーコード検索を中心に小さく始める。  

将来的には、商品情報のDB化、管理画面、複数店舗対応、売場変更管理などに拡張できる。

```text
Urinavi = お客さんが探しやすく、従業員が案内しやすい、店舗向け売り場ナビアプリ
```
